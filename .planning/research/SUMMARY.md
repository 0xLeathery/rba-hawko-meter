# Research Summary

**Project:** RBA Hawk-O-Meter
**Date:** 2025-05-20
**Status:** Research Complete

## Executive Summary

The RBA Hawk-O-Meter is a "Silent Automation" economic dashboard designed to translate complex Australian financial data (RBA rates, ASX futures, economic indicators) into a simple "Traffic Light" gauge for mortgage holders. Unlike commercial fintech apps, it prioritizes a "set-and-forget" serverless architecture using Python ETL scripts on GitHub Actions to fetch data, commit it back to the repo ("Git as Database"), and auto-deploy to Netlify. This ensures zero running costs while maintaining a verifiable history of economic data.

The recommended approach builds a static frontend (Vanilla JS + Plotly.js) that consumes a JSON file generated by the backend. This decoupling allows the backend to handle the complexity of normalizing diverse data sources—specifically tackling the "Money Illusion" and "COVID Outlier" pitfalls via robust statistical methods (Z-scores on ratios, not nominal values)—while the frontend remains dumb and fast. Key differentiators include the "Hawk-O-Meter" gauge and a "Plain English" translator for RBA minutes, avoiding regulatory pitfalls by strictly providing "factual information" rather than "financial advice."

Key risks involve the fragility of scraping non-API sources (CoreLogic/NAB) and the potential for statistical distortion from economic outliers. Mitigation strategies include prioritizing robust libraries like `readabs`, implementing defensive coding with fallbacks, and using rolling windows for Z-score calculations to handle structural economic breaks.

## Key Findings

### Technology Stack
*   **Backend:** **Python 3.11+** is the runtime, leveraging **`readabs`** as a critical component for reliable ABS data fetching. **GitHub Actions** provides free orchestration.
*   **Frontend:** **Vanilla JS** is sufficient (no framework overhead needed). **Plotly.js** is the recommended visualization library for its native Gauge/Indicator support.
*   **Data:** **"Git as Database"** pattern. Data is stored as CSVs in the repo, providing free versioning and history.
*   **Infrastructure:** **Netlify** for zero-config static hosting, triggered automatically by git commits from the ETL pipeline.

### Features
*   **Table Stakes:** Current Cash Rate, Historical Rate Chart, Next RBA Meeting Date, Mobile Responsiveness.
*   **Differentiators:** The **"Hawk-O-Meter" Gauge** (visual sentiment synthesis), **Impact Calculator** (personalization), and **ASX Futures** probability.
*   **Anti-Features:** No "Personal Financial Advice" (e.g., "You should fix your rate"), no User Accounts (privacy liability), no direct product comparisons.
*   **Regulatory:** Must strictly adhere to **ASIC RG 244**, using neutral language ("Market Expectation" vs "Prediction") and prominent disclaimers.

### Architecture
*   **Pattern:** **"Silent Automation"**. A weekly cron job triggers a Python ETL script.
*   **Data Flow:** Ingest (Scrape/API) -> Transform (Normalize/Z-Score) -> Persist (Commit to Git) -> Deploy (Netlify).
*   **Decoupling:** The frontend uses **Runtime Fetching** to load a generated `status.json`. It does not contain business logic, only rendering logic.

### Pitfalls
*   **Critical Data Risks:**
    *   **"COVID Outlier" Effect:** Extreme values (e.g., 2020 GDP) can distort Z-score standard deviations, making normal fluctuations look flat. *Solution: Use Median/IQR or rolling windows.*
    *   **"Money Illusion":** Using nominal values ($) in inflationary times creates invalid trends. *Solution: Strictly use ratios or YoY % changes.*
*   **Operational Risks:** Scraper fragility on non-API sites (CoreLogic). *Solution: Defensive coding, `readabs` priority, and fallbacks.*

## Implications for Roadmap

The research suggests a phased approach centered on establishing the data pipeline first, as the frontend is entirely dependent on the quality of the generated JSON.

### Phase 1: The Data Pipeline (Skeleton)
*   **Rationale:** The "Git as Database" pattern is the core foundation. Reliability here is prerequisite for any UI.
*   **Delivers:** Automated GitHub Action, `readabs` integration, reliable CSV storage of RBA rates.
*   **Key Features:** `ingest.py`, GitHub Workflow setup, CSV history schema.
*   **Pitfalls to Avoid:** Scraper fragility (focus on reliable RBA/ABS sources first).

### Phase 2: The Core Dashboard (MVP)
*   **Rationale:** Delivers the "Table Stakes" value. Users can answer "What is the rate?" and "When is the meeting?".
*   **Delivers:** Live URL on Netlify, basic visualization of Phase 1 data.
*   **Key Features:** Vanilla JS setup, `status.json` contract, Current Rate display, Historical Chart (Plotly).
*   **Pitfalls to Avoid:** Mobile responsiveness issues; maintain separation of concerns (logic in Python, rendering in JS).

### Phase 3: The Hawk-O-Meter (Differentiation)
*   **Rationale:** Transforms the tool from a generic tracker to the unique "Hawk-O-Meter". Requires the statistical engine.
*   **Delivers:** The proprietary "Hawk Score" and Gauge visualization.
*   **Key Features:** `process.py` (Z-score logic), ASX Futures integration, Gauge UI.
*   **Pitfalls to Avoid:** **Critical:** "COVID Outlier" and "Money Illusion" mathematical errors.

### Phase 4: Calculator & Compliance (Refinement)
*   **Rationale:** Adds engagement and ensures regulatory safety before public marketing.
*   **Delivers:** Impact Calculator, "Plain English" tooltips, full compliance audit.
*   **Key Features:** Mortgage Calculator (JS), Legal Disclaimers, Source citations.
*   **Pitfalls to Avoid:** Regulatory guardrails (ASIC RG 244).

## Research Flags

*   **Needs Research (Phase 3):** The specific algorithm for the "Hawk Score" (weighting of CPI vs Unemployment vs Futures) is a gap. We have the *mechanism* (Z-scores) but not the *weights*. This requires a "Data Calibration" task.
*   **Standard Pattern (Phase 1 & 2):** The "Silent Automation" and Vanilla JS patterns are well-understood and low risk.
